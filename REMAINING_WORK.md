Covers verifying each app is listening, DNS setup, nginx install + reload, SSL issuance, and self-serve verification commands (`dig`, `curl -sI`) that isolate whether the problem is DNS, nginx, or the app itself. NOTE: originally written against the old subdomain scheme (`admin.<tenant>.onedexo.com`/`portal.<tenant>.onedexo.com`, which needed a two-level wildcard); the doc has since been updated to the current path-based scheme (`<tenant>.onedexo.com/admin`, `<tenant>.onedexo.com/portal`), which only needs a standard single-level wildcard cert.
T. **Fixed a serious signup/verification bug** — asked "will signup/forgot-password work properly now?" and traced both flows end to end. Forgot-password was already correct (token-based, tenant-agnostic, canonical `/reset-password` page on platform-web works for any user type). Signup was **not**: `register()` generated an unrelated random string as the "verification token," but `verifyEmail()` expects a signed JWT with `type: 'email_verification'` — the random string could never pass `jwtService.verify()`, so verification would always fail. Compounding it: the verification email was never actually sent at all (only the welcome email was), and **no `/verify-email` page existed anywhere in the frontend**. Since `login()` hard-blocks `pending_verification` accounts in production, this meant **every real signup in production would have been permanently locked out with no way to verify**. Fixed: `generateVerificationToken()` now produces the correct JWT shape; `register()` sends a real verification email via a new `TenantMailService.sendVerificationEmail()`; `resendVerificationEmail()` now actually sends instead of a commented-out stub; built `apps/platform-web/app/verify-email/page.tsx` (auto-verifies on load, same "canonical page works for any user type" pattern as reset-password). Also closed a related leak: `resendVerificationEmail` was unconditionally returning the raw verification token in the API response (even in production) — bypassable account verification without ever receiving the email — now gated to non-production like the equivalent `resetToken` field. Also fixed two stale `localhost:3000` fallback URLs (platform-web actually runs on 3001; masked in practice since `FRONTEND_URL` is set correctly in `.env`, but wrong as a fallback).
